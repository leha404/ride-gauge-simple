# feature

# Правило: Как добавить новую фичу

## Цель

- Упростить и стандартизировать процесс добавления новой функции в проект.

## Шаги

1. **Краткое ТЗ**
   - Опиши: цель фичи, ограничения, UX-флоу, какие файлы/сервисы будут затронуты.
   - Пример:
     ```
     - Цель: добавление логгирования расходов топлива
     - Ограничения: UI общий для web/mobile, без сторонних зависимостей
     - UX flow: пользователь вводит данные -> данные сохраняются -> отображаются в истории
     - Тронутые файлы: src/state/fuel.tsx, src/pages/expenses.tsx
     ```

2. **Архитектурный план**
   - Проверь карту архитектуры (`docs/project-map.md`)
   - Спроектируй, где и как должны появиться новые сущности/модули.

3. **Бизнес-логика**
   - Логику реализуй в соответствующем state-файле или доменных helper-ах (например, `src/state/fuel.tsx`)
   - UI-компонентам запрещено включать бизнес-логику вне крайней необходимости.

4. **Персистентность**
   - Все операции с хранением выноси в `src/lib/storage.ts` либо в сервис-репозиторий рядом.

5. **Реализация**
   - Имплементируй по составленному плану, делая атомарные коммиты.

6. **Проверка**
   - Запусти `npm run lint` и `npm run test`
   - Удостоверься, что не сломаны другие модули.

7. **Документация**
   - Опиши новую фичу коротко в README или wiki
   - Обнови архитектурную карту при необходимости

## Out of scope

- Не добавляй функции без выделенного под них места в архитектуре.
- Не реализуй бизнес-логику напрямую в UI.
